分析自动配置

  1. SpringBoot在启动的时候从类路径下的META-INF/spring.factories中获取EnableAutoConfiguration指定的值
  2. 将这些值作为自动配置类导入容器 ， 自动配置类就生效 ， 帮我们进行自动配置工作；
  3. 整个J2EE的整体解决方案和自动配置都在springboot-autoconfigure的jar包中；
  4. 它会给容器中导入非常多的自动配置类 （xxxAutoConfiguration）, 就是给容器中导入这个场景需要的所有组件 ， 并配置好这些组件
  5. 有了自动配置类 ， 免去了我们手动编写配置注入功能组件等的工作；

在这里我们以HttpEncodingAutoConfiguration(Http编码自动配置)为例来讲解

  • 一句话总结 ：根据当前不同的条件判断，决定这个自动配置类是否生效！

    • 一但这个配置类生效；这个配置类就会给容器中添加各种组件；

    • 这些组件的属性是从对应的properties类(这里的ServerProperties)中获取的，这些类里面的每一个属性又是和配置文件绑定的；

    • 所有在配置文件中能配置的属性都是在xxxxProperties类中封装着；

    • 配置文件能配置什么就可以参照某个功能对应的这个属性类

在SpringBoot中，有很多的xxxConfiguration帮助我们进行拓展配置

Configuration（配置类）&Properties（配置属性）&AutoConfiguration（自动配置）

一、Configuration（配置类）

1. 定义与作用

• 核心注解：@Configuration 标记的类表示配置类，用于定义 Spring 容器管理的 Bean。
• 典型场景：
• 显式声明 Bean（如第三方库组件或复杂对象）。
• 替代传统 XML 配置，通过 @Bean 注解方法返回对象。

• 与 @Component 的区别：
@Configuration 是更明确的配置声明，通常结合条件注解（如 @ConditionalOnClass）实现精细化控制，而 @Component 用于通用组件扫描。

二、Properties（配置属性）

1. 外部化配置

• 核心注解：@ConfigurationProperties 将配置文件（如 application.yml）中的属性类型安全地绑定到 Java 对象。



• 优势：
• 支持松散绑定（如 my-prop 对应 myProp）。
• 可嵌套复杂类型（对象、集合、Map）。

2. 与 @Value 的对比

• @Value 适用于单个属性注入，而 @ConfigurationProperties 适合批量属性管理和结构化配置。

三、AutoConfiguration（自动配置）

1. 核心机制

• 触发条件：@EnableAutoConfiguration（隐含在 @SpringBootApplication 中）启用自动配置。
• 实现原理：

  1. 扫描依赖：根据类路径中的 Jar 包（如 spring-boot-starter-web 包含 Tomcat）判断需要加载的模块。
  2. 条件注解：通过 @ConditionalOnClass、@ConditionalOnMissingBean 等动态决定是否创建 Bean。
  3. 配置加载：读取 META-INF/spring.factories 或 AutoConfiguration.imports 文件中的自动配置类列表。

2. 自动配置示例

• 协作关系：自动配置类依赖 DataSourceProperties 读取配置文件中的数据库连接参数。

四、三者的协作关系

机制 协作方式
AutoConfiguration 通过条件注解和类路径扫描，动态加载 @Configuration 类并注入 @ConfigurationProperties 绑定的参数。
Configuration 提供 Bean 定义模板，自动配置类通过 @Bean 方法注册默认组件，用户可覆盖。
Properties 为自动配置类提供外部参数输入（如数据库 URL、端口），支持动态覆盖默认值。

五、自定义配置与覆盖策略

  1. 覆盖自动配置：
    • 定义同名 @Bean（通过 @ConditionalOnMissingBean 确保优先级）。
    • 通过 application.yml 修改配置属性（如 server.port=8081）。
  2. 排除自动配置：
  3. 自定义 Starter：
    创建 META-INF/spring.factories 并注册自定义自动配置类，实现模块化复用。

六、总结

• Configuration：显式定义 Bean，替代 XML 配置。
• Properties：类型安全地管理外部参数，支持复杂结构。
• AutoConfiguration：基于依赖和条件自动装配组件，减少手动配置。
三者协同工作，使 Spring Boot 实现“约定优于配置”的理念，显著提升开发效率。

注意点：

  • WebMvcConfigurer 接口是 Spring MVC 为开发者提供的标准化扩展入口，用于干预框架的核心流程，必须通过接口实现以符合框架的设计规范

    WebMvcConfigurer 是 Spring MVC 框架中用于扩展 MVC 核心功能的标准接口，其设计目的是通过统一的接口方法为开发者提供对 Spring MVC 底层行为的精细化控制。

    例如，配置拦截器（addInterceptors）、跨域规则（addCorsMappings）、静态资源处理（addResourceHandlers）等，均需通过实现该接口并重写对应方法实现。

    Spring MVC 的自动配置类（如 WebMvcAutoConfiguration）通过 DelegatingWebMvcConfiguration 动态收集所有 WebMvcConfigurer 实现类的配置方法，合并后统一生效

  • Druid 配置属于通用组件的替换与参数注入，直接通过 @Bean 和属性绑定即可完成，无需依赖特定接口。两者的差异体现了 Spring 框架对不同场景的扩展性设计策略。(像Swagger也是)

Shrio

Apache Shiro 是一个强大易用的 Java 安全框架，提供了认证、授权、加密和会话管理、web集成，缓存等功能，对于任何一个应用程序，Shiro 都可以提供全面的安全管理服务，并且相对于Spring Security要简单很多。

  • 三个核心组件：Subject, SecurityManager 和 Realms.

    Subject：即“当前操作用户”。但是，在Shiro中，Subject这一概念并不仅仅指人，也可以是第三方进程、后台帐户（Daemon Account）或其他类似事物。它仅仅意味着“当前跟软件交互的东西”。

    Subject代表了当前用户的安全操作，SecurityManager则管理所有用户的安全操作。

    SecurityManager：它是Shiro框架的核心，典型的Facade模式，Shiro通过SecurityManager来管理内部组件实例，并通过它来提供安全管理的各种服务。

    Realm： Realm充当了Shiro与应用安全数据间的“桥梁”或者“连接器”。也就是说，当对用户执行认证（登录）和授权（访问控制）验证时，Shiro会从应用配置的Realm中查找用户及其权限信息。校验用户数据。
